# Recommended Feature — Design Spec

**Date:** 2026-07-22
**Status:** Draft (pending implementation-plan review)
**Project location:** `C:\Users\Wesley Vos\dev\watchlist-app`

## 1. Purpose

A third screen in the app, reached by a "Recommended" nav button, that suggests
films and series the owner might want to watch. Suggestions are generated by a
large language model server-side, grounded in the owner's **watched titles and
the personal ratings** given to them. It turns the accumulated rating history
into a "what to watch next" list without leaving the app.

Single user, same as the rest of the app. This is a v1 of the feature: LLM
recommendations from rating history only. A later grounded-candidate-pool
enhancement ("Option C", section 11) is designed for but not built.

## 2. Core use cases

- Open the Recommended screen and see the last generated set **instantly** (served from cache, no regeneration on open).
- Read each suggestion: title, year, media type, and a one-line reason it was picked.
- Tap a suggestion to jump into the existing preview/add flow and add it to the Want list in one tap.
- Manually refresh to regenerate a fresh set, with a visible "Generated on [date]" caption showing how current the set is.
- On the very first visit (nothing generated yet), tap once to generate the first set.

Explicitly **not** in scope: automatic/scheduled regeneration, per-title
"dismiss/not interested" feedback, infinite scroll. The owner watches few enough
titles that manual refresh is sufficient.

## 3. Stack additions

Everything sits on the existing stack (Next.js App Router, Prisma, Neon
Postgres, Vercel). This feature adds exactly one external dependency and one
env var:

- **LLM provider:** Google Gemini, free tier, via Google AI Studio.
- **Model:** Gemini Flash (free-tier eligible). The exact model id lives behind a single config constant (`GEMINI_MODEL`) and must be confirmed against the current Google AI Studio free-tier model list at implementation time.
- **Transport:** plain server-side `fetch` against the Generative Language REST API, matching how `tmdb.ts` / `omdb.ts` already call their APIs. No SDK, so there is no SDK-version drift to manage.
- **Auth:** `process.env.GEMINI_API_KEY`. The key is provided by the owner in `.env.local` and in Vercel. The spec and code reference `process.env.GEMINI_API_KEY` only, never a literal key.

The Claude subscription/API is explicitly **not** used for this feature.

## 4. Architecture & data flow

The browser talks only to the app's own server routes. The Gemini call happens
server-side, so the API key never reaches the browser (same principle as
TMDb/OMDb).

```
Open screen:  user opens /recommended → GET /api/recommendations
                                       → most recent cached set from DB (or null) → render instantly

Refresh:      tap Refresh → POST /api/recommendations
                          → build rating history from watched+rated titles
                          → recommend service → provider (Gemini) → raw suggestions
                          → resolve each against TMDb (attach real tmdbId + canonical poster/year)
                          → drop unresolved + drop titles already in the library + dedupe
                          → persist as a new RecommendationSet row
                          → return the resolved set → screen updates

Add a suggestion: tap a card → /preview/{mediaType}/{tmdbId} (existing add flow) → Add to Want
```

Two clean seams make provider-swap and Option C cheap later (section 10, 11):

1. **Provider seam:** a `RecommendationProvider` interface (rating history in, raw suggestions out). Gemini is one implementation behind a `getProvider()` factory. Swapping providers is a one-line factory change; no caller changes.
2. **Service seam:** the `recommend` service owns orchestration (history build, provider call, TMDb resolution, library filtering, caching). The provider stays a pure "LLM in/out" unit, unit-testable without a DB or TMDb.

### Why resolve LLM titles against TMDb in v1

The v1 provider generates titles "blind" from ratings, so it returns plain
title strings, not TMDb ids. Two problems follow: a suggestion cannot link into
the preview/add flow (which needs a `tmdbId`), and the LLM can hallucinate a
title that does not exist. Both are solved by a **resolution step**: after the
provider returns, the service runs each suggestion through the existing TMDb
search to attach a real `tmdbId` (and canonical poster/year). A suggestion that
resolves to no TMDb match is dropped. This is distinct from Option C (which
grounds the candidate pool *before* the LLM call); see section 11.

### Resolution disambiguation discipline (reuse, do not reinvent)

Resolution reuses the **exact** disambiguation discipline the Google Takeout
import already established in `scripts/import-google-takeout.ts`
(`evaluateMatch` + `pickDominantCandidate`); it must not introduce a weaker
matcher. The rules, applied per suggestion:

1. **Strong pre-filters first.** The LLM gives a `year` and `mediaType`; use them to narrow the TMDb search results before anything else (media type must match; year gates to the same/adjacent year). This is a signal the Takeout script did not have and makes matching stricter, not looser.
2. **Close-title match + distinct-target dedupe**, exactly as `evaluateMatch` does. A single surviving distinct target is taken directly.
3. **Dominance tiebreak** when several close matches remain: `pickDominantCandidate`'s vote-count-then-popularity rule with the same `DOMINANCE_RATIO` (10x) and floors. A candidate is accepted only if it dominates by that real margin.
4. **Ambiguous means drop.** If no candidate clearly dominates (the Takeout script's "review" outcome), the suggestion is **dropped**, never guessed. A missing recommendation is fine; a wrong one is not.

Because the shared logic lives in the import script, the implementer should
lift/extract that matching code (or import it) rather than write a parallel
matcher, so the two paths cannot drift.

## 5. What the LLM returns (the shape, nailed down)

The provider must return **structured JSON, not prose.** The Gemini request sets
`responseMimeType: "application/json"` with a `responseSchema` describing an
array of objects. The raw, pre-resolution shape:

```ts
// One suggestion exactly as the model returns it (no tmdbId yet).
interface RawSuggestion {
  title: string;              // canonical title as the model knows it
  year: number | null;        // release / first-air year, best effort
  mediaType: "MOVIE" | "TV";  // enum, validated
  reason: string;             // ONE line, why it fits the owner's taste
}
```

Gemini `responseSchema` (conceptual):

```
{
  type: "array",
  items: {
    type: "object",
    properties: {
      title:     { type: "string" },
      year:      { type: "integer", nullable: true },
      mediaType: { type: "string", enum: ["MOVIE", "TV"] },
      reason:    { type: "string" }
    },
    required: ["title", "mediaType", "reason"]
  }
}
```

### Parsing and validation

`responseSchema` reduces malformed output but does not guarantee it, so the
service still validates. A `parseSuggestions(raw: unknown): RawSuggestion[]`
function (hand-rolled, matching the codebase's existing hand-rolled route
validation, no new validation dependency):

- Reads `candidates[0].content.parts[0].text`, `JSON.parse`s it inside a try/catch.
- Iterates the array and keeps only entries where `title` is a non-empty string, `mediaType` is exactly `"MOVIE"` or `"TV"`, and `reason` is a non-empty string. `year` is coerced to a number or `null`.
- **Drops** malformed entries rather than failing the whole set. If zero valid entries remain, that is treated as a provider failure (section 7).
- `reason` is trimmed and length-capped (e.g. 200 chars) so one bad long line cannot break the layout.

### The resolved, cached, rendered shape

After TMDb resolution, each kept suggestion becomes:

```ts
interface ResolvedSuggestion {
  tmdbId: number;
  mediaType: "MOVIE" | "TV";
  title: string;             // TMDb canonical title
  year: number | null;
  posterUrl: string | null;  // TMDb poster
  reason: string;            // carried through from the model
}
```

`ResolvedSuggestion[]` is exactly what is stored in the cache and what the
screen renders. Cards link to `/preview/${mediaType.toLowerCase()}/${tmdbId}`.

## 6. Data model

A single-user app with one "current" recommendation set, plus a light audit
trail. One new table; no changes to `Title`.

### `RecommendationSet`

| Field         | Type          | Notes                                                        |
|---------------|---------------|--------------------------------------------------------------|
| `id`          | string (cuid) | Primary key                                                  |
| `suggestions` | Json          | Array of `ResolvedSuggestion` (section 5)                    |
| `model`       | string        | Model id used (e.g. the `GEMINI_MODEL` value), for provenance |
| `sourceCount` | int           | How many rated titles the set was generated from             |
| `generatedAt` | DateTime      | Default now(); drives the "Generated on [date]" caption      |

- **Read:** the screen and `GET /api/recommendations` return the most recent row by `generatedAt desc` (or `null` if none exist yet).
- **Write:** each successful generation **inserts a new row.** Older rows are harmless; a prune-to-last-N is a trivial later addition and is not required for v1.
- **Manual only:** rows are created only by an explicit `POST /api/recommendations`. Opening the screen never writes.

Migration is generated with the project's established safe method
(`prisma migrate diff --from-schema ... --to-schema ... --script`), never
`prisma migrate dev`, and applied via `prisma migrate deploy` (which the Vercel
build already runs).

## 7. Error handling (the screen never crashes on a bad LLM response)

Every failure degrades to a clear state; a cached set is always preferred over
an error when one exists.

| Situation | Behaviour |
|-----------|-----------|
| No rated history yet (no WATCHED title has a `myRating`) | `POST` returns **`200` with `{ empty: true }`** (a valid state, not an error) and makes **no** Gemini call. The screen shows a guiding empty state ("Rate some watched titles to get recommendations"). Empty history must not be a non-2xx status: the screen treats every non-2xx as "something failed, keep the cached set," so a 4xx here would wrongly trip the error path. |
| Never generated, but history exists | First-run state with a single "Generate recommendations" button that POSTs. |
| Gemini call fails (non-200, network error) | Route returns `502`. If a cached set exists, the screen keeps showing it with a non-blocking message ("Couldn't refresh. Showing your last set."). If none, an error state with Retry. |
| Gemini times out | Server enforces a timeout (AbortController, e.g. 20s) and returns `504`. Same screen behaviour as a failure. |
| Malformed JSON / schema mismatch / zero valid entries | Treated as a provider failure (`502`), same fallback behaviour. The partial-drop rule (section 5) means a set with *some* valid entries still succeeds. |
| Titles that don't resolve against TMDb | Silently dropped during resolution. If **every** suggestion fails to resolve, the generation is treated as a failure (`502`). |
| All resolved titles already in the library | The set can legitimately come back small or empty; the screen shows a gentle "Nothing new to suggest right now, try refreshing later" rather than an error. |

The `POST` route wraps generation in try/catch and always returns structured
JSON; it never throws to the client. The screen treats any non-2xx as "keep the
cached set, surface a soft message."

## 8. How a recommendation connects to the rest of the app

Each `ResolvedSuggestion` carries a real `tmdbId` and `mediaType`, so a card
links straight into the **existing** preview/add flow:
`/preview/{mediaType-lowercased}/{tmdbId}`. That page already fetches full
details and offers "Add to Want" / "Add to Watched." No new add path is built;
the recommendation screen is just another entry point into it.

Because suggestions are external titles not yet in the DB, resolution also
checks each against the library (by `tmdbId` + `mediaType`) and filters out
anything already on a list, so the screen only ever proposes genuinely new
titles.

## 9. Screens & navigation

- **Nav entry:** a "Recommended" button on the Home screen header area (consistent with the existing "+ Add" / search entry vocabulary), routing to `/recommended`. Exact placement is finalized in the polish task with an on-device check.
- **Recommended screen (`/recommended`):** a Back link to Home (same `BackLink` pattern as Search), a header with the "Generated on [date]" caption and a Refresh button, and a grid/list of suggestion cards (poster, title, year, media type, one-line reason). States: loading/generating, first-run, empty (no rated history), populated, and soft-error (populated + message).

Visuals reuse the app's existing card and `meta` vocabulary; no new design
language. Poster tiles reuse the same aspect and styling as the Home grid.

## 10. Provider abstraction (swap-ready)

```
src/lib/recommend/
  types.ts     RatedTitle, RawSuggestion, ResolvedSuggestion,
               RecommendationRequest, RecommendationProvider, RecommendationSet
  provider.ts  RecommendationProvider interface + getProvider() factory (swap point)
  gemini.ts    GeminiRecommendationProvider (fetch + prompt + parse/validate)
  service.ts   orchestration: buildHistory, generate, resolve, filter, persist, getLatest
```

```ts
interface RatedTitle { title: string; year: number | null; mediaType: "MOVIE" | "TV"; myRating: number | null; }

interface RecommendationRequest {
  history: RatedTitle[];
  // Option C (section 11) adds an optional grounded pool here, nothing else changes:
  // candidatePool?: CandidateTitle[];
}

interface RecommendationProvider {
  readonly model: string;
  recommend(req: RecommendationRequest): Promise<RawSuggestion[]>;
}
```

The provider knows nothing about the DB, TMDb, caching, or HTTP. Swapping to a
different LLM later means writing a new class and returning it from
`getProvider()`. Nothing else changes.

## 11. Option C (designed for, not built)

**Option C:** before/alongside the LLM call, the server pulls TMDb's
recommendations/similar-titles for the owner's top-rated titles to build a
grounded **candidate pool** of real titles (each already carrying a `tmdbId`).
The LLM then ranks and explains titles from that pool instead of inventing them.

Where it slots in, without gutting anything:

- **Request:** the already-present optional `RecommendationRequest.candidatePool` field carries the pool. v1 leaves it undefined.
- **Service:** a new `buildCandidatePool()` step runs before the provider call and populates the field. The rest of the orchestration is unchanged.
- **Gemini impl:** the prompt conditionally includes the pool and instructs the model to pick from it. When a pool is present, the model can return the pool's `tmdbId`s directly.
- **Resolution:** for pooled suggestions that already carry a `tmdbId`, the TMDb resolution step is a no-op; the existing resolve-by-search path remains the fallback for any non-pooled title.

So Option C is additive: one new service step, one optional request field, one
conditional prompt branch. No interface or table change.

## 12. Privacy

Only the minimum needed for a recommendation leaves the app to Gemini:

- **Sent:** for each watched, rated title, its `title`, `year`, `mediaType`, and `myRating`. That is all.
- **Never sent:** personal notes, the passcode, `DATABASE_URL` or any credentials, environment values, or any account/device identifiers (there are no user accounts anyway).

The prompt is assembled from that whitelist explicitly (not by serializing whole
`Title` rows), so a future field added to `Title` cannot silently leak.

## 13. Secrets & configuration (environment variables)

Adds one variable to the existing set:

- `GEMINI_API_KEY` — Google AI Studio key, read via `process.env.GEMINI_API_KEY`.

Added to `.env.example` (empty), `.env.local` (real value, gitignored), and
Vercel (all environments). Never hardcoded, never committed. Per the project's
Vercel lesson, it is added as a normal variable, not marked "Sensitive."

## 14. Scope

**In v1:** the `/recommended` screen + nav entry; `GET`/`POST /api/recommendations`;
the provider-abstracted recommend service with a Gemini implementation; TMDb
resolution + library filtering; single-table caching with a "Generated on"
caption; manual refresh only; all failure states; privacy whitelist.

**Designed for, deferred:**

- **Option C** grounded candidate pool (section 11).
- **"Not interested" / dismiss:** a per-suggestion action that hides a title and keeps it out of future sets. Planned, not built in v1. **No v1 schema change is needed to leave room for it:** resolution already filters suggestions against the library by `tmdbId` + `mediaType` (section 8), so adding a dismiss later is the same shape, a small separate table of dismissed `tmdbId` + `mediaType` filtered at the same step. So v1 stays un-over-engineered (no speculative column now) while the later addition is a clean, additive change.
- **Prune-to-last-N** of `RecommendationSet` rows.
